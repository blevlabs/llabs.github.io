Deep Dive: Embeddings

How Trajectory Embeddings
Actually Work

Standard systems embed each memory entry in isolation. We embed each step in the context of the full causal chain — the same breakthrough that transformed document retrieval, applied to agent reasoning.

Isolated Embeddings

How everyone else does it
Each memory entry embedded independently. “Agent concluded X” — no context for what reasoning led to that conclusion.
Retrieval finds semantically similar outcomes. Not similar reasoning patterns or causal chains. You find what happened, not why.
Like embedding a document chunk in isolation — it loses the surrounding context. “Revenue grew 3%” doesn’t know it’s about ACME Corp in Q2 2023.
Can’t propagate new insights backward — no mechanism to find related past experiences by causal pattern.

Trajectory Embeddings

The LLABS approach
Each step embedded in context of the full causal chain. The model sees the entire trajectory and injects global context into each step’s embedding.
Retrieval can target similar decision patterns and failure modes — not just similar outcomes. Useful when the process matters as much as the result.
Four named vectors per experience: reasoning trace, action sequence, environment response, contextualized session embedding. Each queryable independently.
Enables memory-level adaptation: attach new lessons to related past trajectories and make them available in future decisions. The underlying model is unchanged.
The Architecture

How Trajectory Embeddings
Are Structured

Each agent iteration is represented across multiple dimensions rather than a single flat memory vector. This makes it possible to retrieve similar decision patterns and failure modes — not just similar outcomes.

Standard Memory
m_i = f(step_i)

Each memory entry embedded in isolation. The vector represents what happened — but not why it happened, what alternatives were considered, or how the environment shaped the result.

score(q, m_i) = q · m_i
→ retrieves similar outcomes
→ misses similar failure modes
→ misses similar decision patterns
Trajectory Representation
t_i = {v_R, v_A, v_E, v_C}

Each iteration represented across four dimensions. Retrieval can target any combination — by how the agent approached the problem, what it tried, what followed, or the session-level context.

score(q, t_i) = combine(sim(q_R, v_R), sim(q_A, v_A),
                     sim(q_E, v_E), sim(q_C, v_C))
The combine function can be tuned, learned, or reranked.
v_R — Reasoning Trace

embed(reasoning_trace)

The agent’s explicit working rationale, intermediate hypotheses, or decision framing. An approximation of how the agent approached the problem — not a guaranteed record of true internal causality.

v_A — Action Sequence

embed(action_sequence)

Every tool call, parameter, and execution step. Linked to the reasoning that triggered it. Enables retrieval by what the agent actually did, independent of outcome.

v_E — Environment Response

embed(env_response)

The observed consequences: tool outputs, user reactions, errors, state changes. Enables retrieval by what happened as a result — useful for finding similar failure modes and recurring patterns.

v_C — Contextualized

contextualize(step_i, session)

The step re-embedded in the context of the surrounding session. The direct descendant of contextualized chunk embeddings — captures cross-step dependencies invisible to isolated embeddings.

Trajectory embeddings build on contextualized chunk embeddings by treating each step as a session-aware unit, then decomposing that unit into reasoning, action, outcome, and contextual representations for more precise retrieval over recurring decision patterns.

The Mechanics — How Embeddings Are Computed and Searched

What is an embedding?

An embedding model takes text and outputs a vector — a list of numbers, typically 1,000–2,000 dimensions. Similar meanings land near each other in that space. Similarity is measured with cosine similarity: vectors pointing in the same direction are similar; vectors pointing in different directions are not.

“Agent analyzed the patent claim” → [0.23, -0.41, 0.87, ...]
“Prior art search methodology” → [0.21, -0.38, 0.84, ...]
cos_sim = 0.97 → highly similar

Computing the four vectors

For each agent iteration, four separate embedding calls are made — each on a different part of the trace:

v_R = embed(reasoning_text)
v_A = embed(tool_calls + params)
v_E = embed(tool_outputs + feedback)
v_C = embed(step_i + session_context)

v_C is computed with the full session as context, so the vector for step 3 “knows” what happened in steps 1, 2, and 4.

How retrieval works

All four vectors are stored in a vector database. At retrieval time, a query is embedded and compared against stored vectors using cosine similarity. The system returns the top-k most similar past iterations.

query = embed(“how did agent approach claim construction?”)
results = vector_db.search(query, dim=“v_R”, top_k=5)
→ 5 past iterations with similar reasoning

Why four vectors instead of one?

A single vector compresses everything into one point in space — you lose the ability to distinguish “similar reasoning, different outcome” from “different reasoning, same outcome.” Four separate vectors let you search along any dimension independently. You can find all past iterations where the agent reasoned similarly about a problem, regardless of what it actually did or what happened next.

The Learning Mechanism

Memory-Level Adaptation:
The Update Loop

When feedback arrives, the goal is to identify related prior trajectories, attach the new lesson to them, and make that lesson available in future decisions. This is memory-level adaptation — not weight-level learning.

The Update Loop
1. f = embed(feedback, local_context)
2. S = retrieve_related(f, threshold=θ)
3. for each t_i in S:
4.   lesson = derive_lesson(t_i, f)
5.   memory_store.update(t_i, lesson)
6. future_context.inject(S, lessons)

The underlying model is unchanged. This is inference-time adaptation via structured memory — faster to update than retraining, more traceable than weight changes.

Honest Comparison
RLHF / DPO: modifies model weights — broader generalization, but requires many examples and compute. Slower to update. More reliable at scale.
Rules: fast to add, but requires explicit enumeration. Doesn’t generalize beyond exact matches without additional classifiers.
Trajectory: faster to update than retraining, more interpretable. If retrieval is good enough, a lesson from one episode can improve how related episodes are handled in the future.
The value depends on retrieval quality, lesson extraction, and memory hygiene. Without those, the system either misses relevant patterns or generalizes the wrong lesson.
θ

Retrieval Threshold

Similarity in trajectory vector space. Cross-domain transfer is fragile — the value depends heavily on embedding quality and how well the retrieval system is tuned.

∂

Lesson Extraction

Deriving a useful lesson from the comparison of feedback to past trajectory. The hardest part — requires careful design to avoid overgeneralizing one correction.

⚙

Memory Hygiene

Without pruning, conflict resolution, and uncertainty tracking, the memory store can accumulate drift, redundant traces, and bad abstractions over time.

→

Practical Scope

A practical path to faster adaptation with less retraining and more traceable updates — not a substitute for model training, but a complement to it.

The Mechanics — What Each Step of the Update Loop Actually Does

Step 1: embed(feedback, local_context)

The feedback text is embedded together with the local context of the current task — what the agent was doing when the feedback arrived. This produces a vector that represents not just “what was said” but “what was said about this kind of task.” Embedding both together means the retrieval step will find past iterations that are similar in both content and task context.

Step 2: retrieve_related(f, threshold=θ)

The feedback vector is compared against all stored trajectory vectors using cosine similarity. Any past iteration with similarity above threshold θ is included in set S. θ is a tunable parameter — higher means fewer, more precise matches; lower means broader but potentially noisier retrieval. This is standard approximate nearest-neighbor search over a vector database.

S = [t_i for t_i in trajectory_store
     if cos_sim(f, t_i) > θ]
→ returns set of related past iterations

Steps 3–5: derive_lesson and update

For each retrieved trajectory, the system compares the feedback to what the agent did in that past iteration. A lesson is derived — a structured annotation describing what should have been done differently. This lesson is stored alongside the original trajectory, not replacing it. The original trace is preserved; the lesson is an addendum.

lesson = {
  “original_trace”: t_i,
  “correction”: derived_from_feedback,
  “confidence”: cos_sim(f, t_i)
}
memory_store.attach(t_i.id, lesson)

Step 6: future_context.inject(S, lessons)

When the agent starts a new task, the system searches its trajectory store for relevant past iterations. If any retrieved iterations have associated lessons, those lessons are prepended to the agent’s context window before it begins work. The agent doesn’t receive the lesson as a command — it receives it as context, the same way a human might read a case study before starting a similar project.

new_task_context = [
  relevant_past_traces,
  associated_lessons,
  current_task_instructions
]
agent.run(context=new_task_context)

Organizational Intelligence

Reconciling Conflicting
Organizational Feedback

Every organization has people who see the same problem differently. Standard systems fail here — they get conflicting directives and either average them or oscillate unpredictably. We store the context of where feedback came from.

Standard Agent Systems

The failure mode
Head attorney says: “Read my book. This is how to approach patent law.” Agent stores this as a directive.
Junior patent lawyer says: “I know what the book says, but here’s how I actually approach it in practice.” Agent stores this as a conflicting directive.
Result: conflicting directives the agent must balance. It either averages them, picks one, or oscillates between them unpredictably depending on which was stored most recently.
No way to understand why two people see the same thing differently. No context for which approach applies in which situation.

Trajectory-Based Memory

The LLABS approach
Head attorney’s feedback stored with full context: what work led to it, what the agent was doing, what the attorney’s reasoning was, what task type it applied to.
Junior lawyer’s feedback stored with its own context: different task type, different complexity level, different application domain, different constraints.
Result: agent understands both perspectives and the context in which each applies. Not conflicting directives — complementary expertise for different situations.
When a new task arrives, the agent retrieves the most relevant causal pattern and applies the appropriate perspective. Context-aware, not rule-based.
